# 详细执行流程

本文档描述 Paper Formatter Skill 的详细执行流程，供 AI 代理在执行任务时参考。

## 流程总览

```
用户输入 → 需求理解 → 文件确认 → 格式检测 → 需求细化 → 执行修改 → 结果验证 → 交付
```

## Step 1: 需求理解

当用户触发 Skill 后，首先理解用户意图：

1. **识别任务类型**：
   - 全文格式修改（最常见）→ 继续正常流程
   - 局部调整（只改标题、只调页边距等）→ 构建最小化配置
   - 模版应用（"和这篇格式一样"）→ 使用 --template 模式
   - 引用格式转换 → Phase 2 功能，当前告知用户尚未支持

2. **提取格式参数**：
   从用户描述中提取关键参数：
   - "小四宋体" → body_font: name_cn="宋体", size_pt=12
   - "1.5倍行距" → paragraph: line_spacing=1.5
   - "A4纸" → page: paper_size="A4"
   - "页边距3厘米" → page: margin_*_cm=3.0

## Step 2: 文件确认

1. 确认用户要处理的文件路径
2. 如果用户提到"上传"但未提供文件路径，询问文件位置
3. 确认文件格式（.docx / .tex / .md / .pdf）

## Step 3: 格式检测

运行格式检测脚本：
```bash
python <skill_dir>/scripts/detect_format.py <文件路径> --json
```

向用户展示格式摘要：
- 当前纸张大小和页边距
- 使用的字体和字号
- 标题样式
- 段落格式

## Step 4: 需求细化

根据检测结果，与用户确认：

1. 如果检测结果与用户预期不符（如用户说要改但检测显示已经是对的），提醒用户
2. 如果用户的需求不完整（如只说"改格式"但没说具体要求），引导用户补充：
   - 你可以问："需要按什么标准来改？比如国标学位论文、APA、IEEE？"
   - 或者："有参考的模版文档吗？"
3. 对于格式检测中发现的不一致（如正文混用了多种字体），向用户指出

## Step 5: 执行修改

### 模式 A：预设格式
```bash
python <skill_dir>/scripts/format_docx.py <输入> <输出> --preset <preset_name>
```
适用于用户需求匹配已知标准（gb-thesis, ieee, apa）

### 模式 B：JSON 配置
1. 将用户需求整理为 JSON 格式配置文件
2. 用 Write 工具写入临时 JSON 文件
3. 执行：
```bash
python <skill_dir>/scripts/format_docx.py <输入> <输出> --config <tmp_config.json>
```

### 模式 C：模版应用
```bash
python <skill_dir>/scripts/format_docx.py <输入> <输出> --template <模版文件>
```

### 输出文件命名规则
- 默认：`原文件名_formatted.docx`
- 如果用户指定了输出路径，使用用户指定的路径

## Step 6: 结果验证

1. 检查脚本执行输出是否成功
2. 确认输出文件已生成
3. 向用户报告修改摘要

## Step 7: 交付

向用户交付的信息：
1. ✅ 修改完成确认
2. 📂 输出文件路径
3. 📊 修改了哪些内容
4. 💾 备份文件路径（如需恢复）

## 错误处理

| 问题 | 处理方式 |
|------|----------|
| python-docx 未安装 | 运行 `scripts/install_deps.py` |
| 文件不存在 | 请用户确认路径 |
| 文件不是 .docx | 说明当前版本仅支持 .docx |
| 格式配置不完整 | 引导用户补充缺失参数 |
| 修改后格式不对 | 从备份恢复，调整配置后重试 |
